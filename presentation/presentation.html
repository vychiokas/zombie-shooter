How PRP Works

By Rasmus Widing (@Wirasm) · Ralph loop by Geoffrey Huntley · github.com/Wirasm/PRPs-agentic-eng

What is the Ralph Loop?

  • Named after Ralph Wiggum — tries hard, doesn't always nail it
  • Runs the plan task by task, autonomously
  • After each task: runs the validation command
  • If it fails: debugs, fixes, and retries
  • Keeps looping until all checks pass
  • You check in at the end, not during

"I'm helping!"

What Actually Happens

  • Creates a git branch before touching anything
  • Explores your codebase for existing patterns to copy
  • Proposes a plan with file and line references
  • Asks clarifying questions before proceeding
  • Plan fits your code — not generic tutorials

What It Feels Like

Like handing a task to a mid-level dev.

  • Comes back with a plan before writing a single line
  • Asks when something is unclear — doesn't guess
  • Doesn't always get it right the first time

Project Structure

Code Design

PATTERNS USED

  • Template Method — Scene base class defines handle_event / update / draw interface
  • Flyweight — sprites loaded once at module level, shared across all entity instances
  • Separation of concerns — systems are stateless functions, entities hold only data and self-rendering

SOLID IN PRACTICE

  • S — each class has one job: Spawner spawns, Collisions resolves, Animation animates
  • O — new zombie variants added without touching existing code
  • L — MenuScene, PlayScene, GameOverScene swap into Game interchangeably
  • D — Game depends on Scene interface, not concrete scenes

The Game Loop

Every frame (60 fps)
▼
dt = clock.tick(60) / 1000
▼
scene.handle_event(event)
▼
scene.update(dt)
▼
scene.draw(screen)
▼
pygame.display.flip()
handle_event — keyboard, mouse, quit
update(dt) — move entities, run systems, check collisions
draw — render everything to screen in order
dt — delta time keeps movement frame-rate independent

When NOT to Use PRP

  • One-liners and quick fixes — overhead isn't worth it
  • Exploratory work where you don't know what you want yet
  • When the problem is understanding, not building
  • UI tweaks, copy edits — just do it
  • When you need to deeply understand the code yourself

PRP trades understanding for speed — sometimes you need both

A Programmer's Take

AI writes the code. You're still responsible for it.

  • Read and understand everything it produces
  • Wrong direction caught early is cheap — caught late is painful
  • Review the plan AND the code after

Is PRP Ideal?

…right?

No.

Claude wrote the bugs too.

  • Explosion kills don't remove zombies — they quietly persist until a bullet finishes them
  • Shotgun can trigger chain explosions 3× from a single Exploder death
  • Reviewing the plan isn't the same as reviewing the code
  • Claude found these bugs when asked to review

My Verdict

Compressing the first week of a project into an afternoon.

  • Should work with anything, not just games
  • Clean enough to build MVPs from scratch or even continue ongoing work
  • If you want something production ready - you'll need to do the work
  • Saves time trading quality
  • You are the owner not the agent
